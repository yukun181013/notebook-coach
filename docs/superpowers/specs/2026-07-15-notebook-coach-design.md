# Notebook Coach 设计规格

- 状态：已由用户批准，等待书面规格复核
- 日期：2026-07-15
- 目标赛道：OpenAI Build Week — Education
- 产品形态：Codex Skill + 本地 Python 工具链

## 1. 背景与产品目标

学习者经常拿到一个能够运行、部分运行或已经报错的 Jupyter Notebook，却不知道自己究竟在哪个概念上理解不完整。普通代码检查器偏向修复代码，普通问答又缺少可复查的单元格证据。Notebook Coach 的目标是把 Notebook 变成一个短反馈循环：

1. 对 Notebook 做静态诊断；
2. 用具体单元格作为证据解释代码问题和概念缺口；
3. 生成两个针对性练习；
4. 学习者修改原 Notebook 后复检；
5. 展示问题是否解决、学习证据如何变化，以及下一步最值得学习的内容。

首版聚焦 Python 与大模型课程 Notebook，尤其是包含 Transformer、attention、提示工程、RAG 或基础机器学习代码的教学材料。

## 2. 范围与非目标

### 2.1 首版范围

- 以 Codex Skill 的形式运行，并明确要求在 GPT-5.6 Codex 会话中使用。
- 默认只做静态分析，不执行用户代码。
- 输出独立的诊断报告、挑战 Notebook 和复检报告。
- 在用户明确确认后，可对临时副本进行受限执行。
- 不修改用户的源 Notebook。
- 不需要 OpenAI API Key，也不产生独立的 OpenAI API 账单；它仍会消耗用户现有的 Codex/ChatGPT 使用额度或活动 credits。
- 官方支持 macOS、Linux、Python 3.11，以及 Codex CLI 或 Codex 桌面应用。

### 2.2 非目标

- 不做网页应用、账号系统、云后端或数据库。
- 不做完整 LMS、课程管理或长期学生画像。
- 不自动修复源 Notebook，也不替学习者填写挑战答案。
- 不声称临时副本和超时机制是操作系统级沙箱。
- 首版不承诺原生 Windows 支持；WSL 仅标注为未验证。
- 不把 Learning Evidence Score 描述为学习者能力、成绩或认证结论。

## 3. 核心用户体验

### 3.1 首次诊断

建议调用形式：

```text
$notebook-coach 检查 path/to/course.ipynb
```

系统按以下顺序工作：

1. 预检文件格式、Notebook 规模、kernel 信息、敏感内容与高风险代码模式。
2. 本地 Python 工具生成经过清洗的静态快照，不执行任何单元格。
3. GPT-5.6 基于快照识别知识点、代码错误、概念误解、缺失步骤、可复现性问题和不良实践。
4. 每条实质性判断必须引用具体单元格或明确标记为“证据不足”。
5. 在独立输出目录中生成基线、报告和挑战 Notebook。
6. 若执行能显著提高判断可信度，先向用户说明原因、风险、kernel、路径和超时，再请求一次明确确认。

默认输出目录：

```text
notebook-coach-output/<notebook-name>/
├── baseline.json
├── report.md
└── challenge.ipynb
```

可选执行后增加：

```text
├── execution-log.json
```

### 3.2 复检

建议调用形式：

```text
$notebook-coach 复检 path/to/course.ipynb
```

复检以不可变的 `baseline.json` 为起点，为当前 Notebook 生成新的清洗快照，并输出：

```text
verification.md
```

报告需要说明：

- 哪些原问题已解决；
- 哪些问题仍存在或出现回归；
- 两个挑战分别是否完成；
- Learning Evidence Score 的分项变化；
- 当前最值得继续学习的一个目标。

源文件哈希发生变化是学习者修改后的正常现象。系统只在基线身份无法可靠匹配时请求确认，例如路径或 Notebook 标识完全不同、kernel 语言变化、基线缺失或用户指定了另一份 Notebook。

## 4. 架构与职责边界

### 4.1 总体架构

```text
源 Notebook
    │
    ▼
本地预检与静态快照 ──► baseline.json
    │
    ▼
GPT-5.6 教学诊断
    ├──────────────► report.md
    └──────────────► challenge.ipynb
                         │
               学习者修改源 Notebook
                         │
                         ▼
              本地确定性检查 + GPT-5.6 复评
                         │
                         ▼
                  verification.md
```

### 4.2 模块职责

| 单元 | 职责 | 依赖 | 不负责 |
| --- | --- | --- | --- |
| `SKILL.md` | 编排流程、交互确认、教学判断规则、产物要求 | Codex、GPT-5.6、本地脚本 | 直接解析二进制输出或修改源文件 |
| 本地解析脚本 | 校验 Notebook、清洗内容、编号单元格、生成快照和哈希 | Python 3.11、Notebook 解析库 | 概念教学判断 |
| 本地风险扫描脚本 | 识别敏感值与潜在危险执行模式 | Python 3.11 | 宣称完整安全隔离 |
| 临时执行脚本 | 获得确认后复制并执行、控制超时、记录结果 | 本地 kernel 与现有依赖 | 自动安装依赖或执行高风险 Notebook |
| 比较与验证脚本 | 做确定性检查、比较基线与当前快照、校验产物结构 | Python 3.11 | 判断开放式解释的教学质量 |
| GPT-5.6 | 概念诊断、证据化解释、挑战设计、开放式答案复评 | 当前 Codex 会话中的清洗快照 | 直接执行代码或伪造运行结果 |

每个单元通过文件或明确的结构化对象通信，使解析、执行、教学判断和输出验证可以分别测试和替换。

## 5. 数据契约

### 5.1 `baseline.json`

`baseline.json` 是首次诊断时的不可变基线，至少包含：

- schema 版本；
- 源 Notebook 的相对或规范化路径、文件哈希和采集时间；
- kernel、语言和必要元数据；
- 单元格索引、类型、清洗后的 source 与执行序号；
- 文本形式的输出、异常类型和异常摘要；
- 结构性问题，例如执行序号错乱、缺少导入、隐藏状态依赖；
- 已应用的清洗和脱敏记录；
- 是否执行过代码，以及执行结果来自原有输出还是临时执行。

以下内容不得写入模型上下文或基线：

- Base64 图片与附件主体；
- 超过限制的完整大输出；
- 检测到的密钥、token、密码或类似敏感值原文；
- Notebook 无关的本地环境秘密。

大输出应保留类型、原始长度、摘要和截断标记，以便报告诚实说明证据边界。

### 5.2 `report.md`

报告采用固定章节：

1. Notebook 概览；
2. 概念地图；
3. 关键问题与单元格证据；
4. Learning Evidence Score；
5. 推荐挑战；
6. 可选执行结果与限制。

问题记录至少包含严重度、类别、单元格编号、证据、影响和学习建议。没有证据时必须写“证据不足”，不能补写不存在的运行结果或学习者意图。

### 5.3 Learning Evidence Score

总分 100，只衡量当前 Notebook 中可观察到的学习证据：

| 维度 | 权重 |
| --- | ---: |
| 正确性 | 30 |
| 概念完整度 | 30 |
| 可复现性 | 20 |
| 解释与代码清晰度 | 20 |

每次扣分必须绑定具体单元格和可复用规则。分数用于展示同一 Notebook 修改前后的证据变化，不用于跨学生排名，也不代表用户的绝对能力。

### 5.4 `challenge.ipynb`

挑战 Notebook 固定包含两个任务：

1. 一个代码修复或补全任务；
2. 一个概念解释或结果分析任务。

每个任务只提供完成任务所需的最小上下文，并包含：

- 学习目标；
- `TODO`；
- 验收标准或 rubric；
- 与原 Notebook 问题的关联说明。

挑战不得给出答案，也不得复制不必要的大段课程内容。挑战应能在数分钟内完成，以适配演示和快速学习循环。

### 5.5 `verification.md`

复检结合两类证据：

- 本地确定性检查：是否可执行、断言结果、异常、预期输出和结构变化；
- GPT-5.6 教学判断：解释是否正确、概念缺口是否解决、学习证据是否充分。

报告固定包含修改前后分数、已解决问题、剩余问题、回归、挑战状态和一个下一步学习目标。无法执行时不得把静态推断表述成运行事实。

## 6. 安全、隐私与成本边界

### 6.1 预检限制

- 文件必须是合法 `.ipynb` JSON。
- 默认最多处理 200 个单元格。
- 移除附件和大输出后，源文本默认上限约为 150,000 字符。
- 超限时不静默截断，应要求用户选择章节或单元格范围。
- 在进入模型上下文前扫描常见 API Key、密码、token 和私钥模式，并替换为 `[REDACTED]`。
- 扫描 `os.system`、`subprocess`、删除文件、网络访问、shell magic 和其他明显高风险模式。

### 6.2 可选执行协议

只有同时满足以下条件才允许执行：

1. 静态诊断不足以验证关键问题；
2. 风险扫描没有触发阻断；
3. 系统向用户展示源路径、kernel、临时目录、逐单元格超时和总超时；
4. 用户明确确认。

执行只针对临时副本，默认逐单元格超时 30 秒、总超时 120 秒。超时后终止子进程并写入 `execution-log.json`。执行过程不得更改源 Notebook，也不得在源 Notebook 所在目录写入产物。

临时副本、模式扫描和超时不是安全沙箱。执行代码仍继承当前用户权限，因此 README 必须明确要求只执行自己拥有或已经审查的 Notebook；演示只使用仓库附带的安全样例。

### 6.3 隐私与账单说明

Notebook 内容不发送到自建服务，也不需要用户配置 OpenAI API Key。不过，经过清洗的 Notebook 文本会进入当前 Codex/GPT-5.6 会话。项目不能宣传为“完全离线”，也不能宣传为“零算力成本”；准确表述是“无独立 API Key 与 API 账单，使用现有 Codex/ChatGPT 额度或活动 credits”。

## 7. 错误处理

- Notebook JSON 无效：停止并给出可操作的解析错误，不创建误导性报告。
- 依赖缺失：继续静态诊断，列出缺失包，不自动安装。
- 现有单元格报错：将错误作为诊断证据，不让单个错误中止整个流程。
- 执行超时：终止子进程，保留已完成步骤和超时位置。
- 风险扫描命中：阻止自动执行，仍允许静态诊断。
- 证据不足：显式标记，不推测未观察到的行为。
- 基线无法匹配：说明差异并请求确认，不自动比较两份不相关的 Notebook。
- 输出写入失败：先写临时文件再原子替换；不得覆盖已有 `baseline.json`。
- 再次首次诊断同名 Notebook：创建带时间戳或唯一后缀的新运行目录，避免混淆历史基线。

## 8. 测试与验收

### 8.1 自动化验收场景

至少覆盖以下案例：

1. 正常 Notebook 生成全部必需产物，且结构校验通过；
2. 语法错误和已有运行时错误能定位到正确单元格；
3. 依赖缺失时静态诊断仍完成；
4. 无限循环在临时执行中按总时限终止；
5. 常见敏感值在模型上下文和产物中被脱敏；
6. shell、网络或危险文件系统操作阻止自动执行；
7. Base64 与巨大输出不会进入模型上下文；
8. 复检能识别已解决问题、回归和分数变化；
9. 工具运行前后源 Notebook 哈希保持不变；
10. 产物写入中断不会破坏已有基线；
11. 每项扣分都能追溯到单元格和规则；
12. 静态推断与实际执行结果在报告中有明确标签。

### 8.2 安全演示样例

仓库提供 `samples/transformer_attention_buggy.ipynb`，只包含安全、确定且可快速演示的内容：

- 一个可定位的代码错误；
- 一个关于 attention 的概念误解；
- 一个可复现性问题。

样例应能让首次诊断、两个挑战和复检在五分钟内完成。仓库同时保留预生成的 `report.md`、`challenge.ipynb` 和 `verification.md`，使评委无需安装也能理解完整体验。

### 8.3 成功标准

- 新用户从全新 checkout 到生成第一份报告不超过五分钟；
- 默认路径不执行任何 Notebook 代码；
- 源 Notebook 在工具运行中从不被修改；
- 每个关键诊断都有单元格证据或“证据不足”标签；
- 两个挑战都能映射到已发现的问题；
- 复检展示可解释的前后变化，而非只给一个新分数；
- 核心演示可在三分钟视频内讲清楚。

## 9. README 与评委体验

README 至少包含：

- 一分钟安装步骤；
- “不需要 API Key”的准确成本说明；
- GPT-5.6、Codex 与本地 Python 脚本的职责；
- 首次诊断和复检命令；
- 支持平台和依赖；
- 执行风险、隐私与限制；
- 自动化测试命令；
- 安全样例与预生成产物链接；
- Codex 如何参与开发和关键产品决策；
- Devpost `/feedback` 所需 Codex Session ID 的记录位置；
- 仓库、安装、平台和测试说明。

评委应能沿着一条最短路径理解项目：打开安全样例，查看预生成诊断，查看两个挑战，再查看复检前后差异。安装不是理解产品价值的前提。

## 10. 三分钟演示脚本

| 时间 | 内容 |
| --- | --- |
| 0:00–0:20 | 学习者面对“能跑但没学懂”的 Notebook 的问题 |
| 0:20–0:45 | 安装并运行首次诊断 |
| 0:45–1:25 | 展示带单元格证据的报告和分项分数 |
| 1:25–1:55 | 展示一个代码挑战和一个概念挑战 |
| 1:55–2:25 | 修改后复检，展示已解决问题与分数变化 |
| 2:25–2:50 | 说明 GPT-5.6、Codex、本地脚本各自负责什么 |
| 2:50–3:00 | 强调无 API Key、源文件不被修改和教育价值 |

视频使用公开 YouTube 链接、有声讲解且少于三分钟。演示应使用附带样例，不临场运行未知代码。

## 11. 关键设计取舍

### 11.1 选择 Codex Skill，而非 Web 应用

Skill 最贴合活动要求，也能直接使用 GPT-5.6 与 Codex，无需另建 API 后端、认证、部署和计费链路。代价是体验依赖 Codex 客户端，无法提供独立网页界面；首版接受这一限制，以换取更低实现风险和更强的提交完整度。

### 11.2 静态分析优先，执行必须确认

完全自动执行更容易展示运行结果，但风险和环境不确定性过高。静态分析优先可保证默认安全和快速响应；受限执行只作为提高证据可信度的可选步骤。

### 11.3 分离报告与挑战 Notebook

把诊断写回源 Notebook 容易污染课程材料，也难以比较修改前后状态。独立报告便于评委阅读，独立挑战 Notebook 则保留可交互的学习体验，同时维持源文件不可变。

## 12. 完成定义

首版只有在以下事项全部满足时才算完成：

- Skill 可从全新 checkout 安装并调用；
- 静态解析、脱敏、风险扫描、产物生成和复检均有自动化测试；
- 可选执行具有确认、临时副本、超时和日志；
- 安全样例可完成完整演示闭环；
- README 准确覆盖安装、平台、成本、隐私和安全边界；
- 三分钟以内公开视频、公开仓库和 Devpost 必填字段准备完毕；
- 使用 GPT-5.6 的 Codex 会话及 `/feedback` Session ID 已记录；
- 提交前在受支持的全新环境完成一次端到端验证。
