# Notebook Coach 设计规格

- 状态：已由用户批准，等待书面规格复核
- 日期：2026-07-15
- 目标赛道：OpenAI Build Week — Education
- 产品形态：Codex Skill + 本地 Python 工具链

## 1. 背景与产品目标

学习者经常拿到一个能够运行、部分运行或已经报错的 Jupyter Notebook，却不知道自己究竟在哪个概念上理解不完整。普通代码检查器偏向修复代码，普通问答又缺少可复查的单元格证据。Notebook Coach 的目标是把 Notebook 变成一个短反馈循环：

1. 对 Notebook 做静态诊断；
2. 用具体单元格作为证据解释代码问题和概念缺口；
3. 生成两个针对性练习；
4. 学习者修改原 Notebook 后复检；
5. 展示问题是否解决、学习证据如何变化，以及下一步最值得学习的内容。

首版聚焦 Python 与大模型课程 Notebook，尤其是包含 Transformer、attention、提示工程、RAG 或基础机器学习代码的教学材料。

## 2. 范围与非目标

### 2.1 首版范围

- 以 Codex Skill 的形式运行，并明确要求在 GPT-5.6 Codex 会话中使用。
- 默认只做静态分析，不执行用户代码。
- 输出独立的诊断报告、挑战 Notebook 和复检报告。
- 在用户明确确认后，可对临时副本进行受限执行。
- 不修改用户的源 Notebook。
- 不需要 OpenAI API Key，也不产生独立的 OpenAI API 账单；它仍会消耗用户现有的 Codex/ChatGPT 使用额度或活动 credits。
- 官方支持 macOS、Linux、Python 3.11，以及 Codex CLI 或 Codex 桌面应用。

### 2.2 非目标

- 不做网页应用、账号系统、云后端或数据库。
- 不做完整 LMS、课程管理或长期学生画像。
- 不自动修复源 Notebook，也不替学习者填写挑战答案。
- 不声称临时副本和超时机制是操作系统级沙箱。
- 首版不承诺原生 Windows 支持；WSL 仅标注为未验证。
- 不把 Learning Evidence Score 描述为学习者能力、成绩或认证结论。

## 3. 核心用户体验

### 3.1 首次诊断

建议调用形式：

```text
$notebook-coach 检查 path/to/course.ipynb
```

系统按以下顺序工作：

1. 预检文件格式、Notebook 规模、kernel 信息、敏感内容与高风险代码模式。
2. 本地 Python 工具生成经过清洗的静态快照，不执行任何单元格。
3. GPT-5.6 基于快照识别知识点、代码错误、概念误解、缺失步骤、可复现性问题和不良实践。
4. 每条实质性判断必须引用具体单元格或明确标记为“证据不足”。
5. 在独立输出目录中生成基线、报告和挑战 Notebook。
6. 若执行能显著提高判断可信度，先向用户说明原因、风险、kernel、路径和超时，再请求一次明确确认。

每次首次诊断都创建独立运行目录。`source-id` 是规范化源路径的短哈希，`run-id` 由 UTC 时间戳和首次文件哈希短值组成：

```text
notebook-coach-output/<notebook-name>-<source-id>/<run-id>/
├── baseline.json
├── report.md
└── challenge.ipynb
```

可选执行后增加：

```text
├── execution-diagnosis-source-A001.json
```

所有执行日志都采用 `execution-<phase>-<target>-<attempt-id>.json`，其中 phase 为 `diagnosis` 或 `verification`，target 为 `source` 或 `challenge`，attempt ID 是 run 内从 `A001` 开始的单调编号。第一次尝试也必须带 ID，任何日志都不得覆盖。

### 3.2 复检

学习者可以修改源 Notebook 来修复原问题，并在运行目录内填写 `challenge.ipynb` 的两个任务。两类修改分别衡量：源 Notebook 决定前后问题和 Learning Evidence Score，挑战 Notebook 决定两个挑战的完成状态。只修改挑战 Notebook 时，挑战状态可以改善，但源 Notebook 分数不会被虚构为提高。

建议调用形式：

```text
$notebook-coach 复检 path/to/course.ipynb --run notebook-coach-output/<name>-<source-id>/<run-id>
```

`--run` 表示自然语言调用中的明确运行目录参数，并不要求额外发布一个独立 CLI。复检从该目录读取不可变的 `baseline.json` 和同目录下由用户填写的 `challenge.ipynb`，再为当前源 Notebook 生成新的清洗快照，并把 `verification.md` 写回同一运行目录。

如果用户省略 `--run`，系统只能按 `baseline.json` 中记录的规范化源路径精确匹配：只有一个候选时可以继续；没有候选时报错；存在多个候选时必须列出 run ID 并让用户选择，不能猜测“最新一次”。

复检输出：

```text
verification.md
```

报告需要说明：

- 哪些原问题已解决；
- 哪些问题仍存在或出现回归；
- 两个挑战分别是否完成；
- Learning Evidence Score 的分项变化；
- 当前最值得继续学习的一个目标。

源文件哈希发生变化是学习者修改后的正常现象。系统只在基线身份无法可靠匹配时请求确认，例如规范化路径不同、kernel 语言变化、基线缺失或用户指定了另一份 Notebook。

## 4. 架构与职责边界

### 4.1 总体架构

```text
源 Notebook
    │
    ▼
本地预检与静态快照
    │
    ▼
GPT-5.6 教学诊断
    ├──────────────► baseline.json
    ├──────────────► report.md
    └──────────────► challenge.ipynb
           │                 │
           │       学习者填写两个挑战
           │                 │
           └──── 学习者修改源 Notebook
                             │
                             ▼
                  选定 run + 本地确定性检查
                         + GPT-5.6 复评
                             │
                             ▼
                      verification.md
```

### 4.2 模块职责

| 单元 | 职责 | 依赖 | 不负责 |
| --- | --- | --- | --- |
| `SKILL.md` | 编排流程、交互确认、教学判断规则、产物要求 | Codex、GPT-5.6、本地脚本 | 直接解析二进制输出或修改源文件 |
| 本地解析脚本 | 校验 Notebook、清洗内容、编号单元格、生成快照和哈希 | Python 3.11、Notebook 解析库 | 概念教学判断 |
| 本地风险扫描脚本 | 识别敏感值与潜在危险执行模式 | Python 3.11 | 宣称完整安全隔离 |
| 临时执行脚本 | 获得确认后复制并执行、控制超时、记录结果 | 本地 kernel 与现有依赖 | 自动安装依赖或执行高风险 Notebook |
| 比较与验证脚本 | 做确定性检查、比较基线与当前快照、校验产物结构 | Python 3.11 | 判断开放式解释的教学质量 |
| GPT-5.6 | 概念诊断、证据化解释、挑战设计、开放式答案复评 | 当前 Codex 会话中的清洗快照 | 直接执行代码或伪造运行结果 |

每个单元通过文件或明确的结构化对象通信，使解析、执行、教学判断和输出验证可以分别测试和替换。

## 5. 数据契约

### 5.1 `baseline.json`

`baseline.json` 是首次诊断时的不可变静态基线，至少包含：

- schema 版本、评分规则版本和 `run_id`；
- 源 Notebook 的相对或规范化路径、文件哈希和采集时间；
- kernel、语言和必要元数据；
- 单元格索引、类型、清洗后的 source 与执行序号；
- 文本形式的输出、异常类型和异常摘要；
- 结构性问题，例如执行序号错乱、缺少导入、隐藏状态依赖；
- 已应用的清洗和脱敏记录；
- 原 Notebook 已保存输出的证据来源标签，例如 `saved_notebook_output`；
- 带稳定 `issue_id` 的首次诊断问题、所属评分维度和严重度。

可选临时执行结果永远不回写 `baseline.json`。它只写入同一 run 的目标隔离执行日志，并通过 `run_id`、目标类型和目标输入哈希建立关联。

以下内容不得写入模型上下文或基线：

- Base64 图片与附件主体；
- 超过限制的完整大输出；
- 检测到的密钥、token、密码或类似敏感值原文；
- Notebook 无关的本地环境秘密。

大输出应保留类型、原始长度、摘要和截断标记，以便报告诚实说明证据边界。

### 5.2 `report.md`

报告采用固定章节：

1. Notebook 概览；
2. 概念地图；
3. 关键问题与单元格证据；
4. Learning Evidence Score；
5. 推荐挑战；
6. 可选执行结果与限制。

问题记录至少包含稳定 `issue_id`、严重度、评分维度、类别、单元格编号、证据、影响和学习建议。没有证据时必须写“证据不足”，不能补写不存在的运行结果或学习者意图。

`report.md` 是可原子更新的展示产物，不是基线。首次静态诊断先生成 revision 1。首次诊断阶段唯一允许执行的目标是源 Notebook 的临时副本。若用户确认，系统写入 `execution-diagnosis-source-<attempt-id>.json`，然后原子生成下一 revision，只更新“可选执行结果”及证据标签，标明哪些静态判断被支持、冲突或仍不确定。该步骤不重算首次评分，不改写首次问题集合，也不改写 `baseline.json` 或 `challenge.ipynb`。若执行结果暴露出会实质改变诊断或挑战的新问题，报告要求用户开启一次新的首次诊断 run，而不是悄悄改变当前基线。

### 5.3 Learning Evidence Score

总分 100，只衡量当前 Notebook 中可观察到的学习证据：

| 维度 | 权重 |
| --- | ---: |
| 正确性 | 30 |
| 概念完整度 | 30 |
| 可复现性 | 20 |
| 解释与代码清晰度 | 20 |

评分规则采用 `rubric_version: "1.0"`。每个问题只能归入一个主要维度，并按严重度扣分：

| 严重度 | 扣分 | 定义 |
| --- | ---: | --- |
| `blocking` | 10 | 阻止 Notebook 完成，或使核心结论无效 |
| `major` | 5 | 核心逻辑、概念或可复现性存在实质缺口 |
| `minor` | 2 | 局部清晰度或实践问题，不推翻主要结果 |

各维度从其满分开始，减去该维度问题的扣分并以 0 为下限；总分是四个维度之和。GPT-5.6 负责依据证据分类问题，Python 脚本只做可重复的算术和结构校验。复检优先沿用原 `issue_id`；已解决问题移除其扣分，新发现的问题获得新 ID。每次扣分必须绑定具体单元格、严重度和规则版本。

分数用于展示同一源 Notebook 修改前后的证据变化，不用于跨学生排名，也不代表用户的绝对能力。

### 5.4 `challenge.ipynb`

挑战 Notebook 固定包含两个任务：

1. 一个代码修复或补全任务；
2. 一个概念解释或结果分析任务。

每个任务只提供完成任务所需的最小上下文，并包含：

- 稳定 `challenge_id`；
- 学习目标；
- `TODO`；
- 验收标准或 rubric；
- 一个或多个 `source_issue_ids`，明确关联原 Notebook 问题。

`run_id`、两个 `challenge_id`、`source_issue_ids` 和初始挑战内容哈希写入 Notebook 的 `metadata.notebook_coach`。用户应直接填写运行目录中的这份挑战 Notebook；复检按元数据定位两个任务，元数据缺失或 run ID 不匹配时停止挑战判定并说明原因。

挑战不得给出答案，也不得复制不必要的大段课程内容。挑战应能在数分钟内完成，以适配演示和快速学习循环。

### 5.5 `verification.md`

复检结合两类证据：

- 本地确定性检查：是否可执行、断言结果、异常、预期输出和结构变化；
- GPT-5.6 教学判断：解释是否正确、概念缺口是否解决、学习证据是否充分。

报告固定包含 run ID、修改前后分数、已解决问题、剩余问题、回归、两个 challenge ID 的状态和一个下一步学习目标。源 Notebook 的问题变化和挑战完成状态必须分开呈现。无法执行时不得把静态推断表述成运行事实。

复检默认先生成静态 `verification.md` revision 1。复检把当前源 Notebook 和已填写的 `challenge.ipynb` 视为两个完全独立的执行目标：

- `source` 目标的执行证据只能影响源问题的解决/回归判断，以及源 Notebook 的证据标签；
- `challenge` 目标的执行证据只能影响代码挑战的完成状态；概念挑战始终由静态文本和 GPT-5.6 教学判断评估；
- 每个目标都要单独完成风险扫描、说明目标路径与哈希，并获得一次明确确认；确认一个目标不授权执行另一个；
- 每个目标在不同的临时目录和子进程中执行，分别使用逐单元格 30 秒、总计 120 秒的限制；
- 日志分别命名为 `execution-verification-source-<attempt-id>.json` 和 `execution-verification-challenge-<attempt-id>.json`。

完成任一已确认目标后，系统原子生成下一版 `verification.md`，只把该目标的运行证据加入对应判断。该步骤不改写 `baseline.json`、`report.md` 或 `challenge.ipynb`，也不得跨目标复用运行证据。

## 6. 安全、隐私与成本边界

### 6.1 预检限制

- 文件必须是合法 `.ipynb` JSON。
- 默认最多处理 200 个单元格。
- 移除附件和大输出后，源文本默认上限约为 150,000 字符。
- 超限时不静默截断，应要求用户选择章节或单元格范围。
- 在进入模型上下文前扫描常见 API Key、密码、token 和私钥模式，并替换为 `[REDACTED]`。
- 扫描 `os.system`、`subprocess`、删除文件、网络访问、shell magic 和其他明显高风险模式。

### 6.2 可选执行协议

只有同时满足以下条件才允许执行某一个明确目标：

1. 静态诊断不足以验证关键问题；
2. 已对该目标单独完成风险扫描且没有触发阻断；
3. 系统向用户展示目标类型、目标路径、输入哈希、kernel、临时目录、逐单元格超时和总超时；
4. 用户明确确认该目标。

执行只针对所选目标的临时副本，默认逐单元格超时 30 秒、每个目标总超时 120 秒。超时后终止该目标的子进程并写入执行日志。日志记录 `run_id`、phase、target、attempt ID、基线哈希、目标输入哈希、临时副本哈希、时间限制、退出状态和结果来源。首次诊断阶段只允许 `diagnosis/source`；复检阶段允许分别选择 `verification/source` 和 `verification/challenge`。

工具自身只在临时目录和 run 目录写入文件，不写源 Notebook 或源目录。每次执行前后都重新计算所选目标的输入哈希；若同一次工具控制的执行期间目标哈希变化，则标记 `target_changed_during_run`，停止对应报告最终化并要求用户检查。学习者在首次诊断结束后、复检开始前主动修改源 Notebook 或填写挑战 Notebook 是预期行为。

临时副本、模式扫描和超时不是安全沙箱。执行代码仍继承当前用户权限，因此 README 必须明确要求只执行自己拥有或已经审查的 Notebook；演示只使用仓库附带的安全样例。

### 6.3 隐私与账单说明

Notebook 内容不发送到自建服务，也不需要用户配置 OpenAI API Key。不过，经过清洗的 Notebook 文本会进入当前 Codex/GPT-5.6 会话。项目不能宣传为“完全离线”，也不能宣传为“零算力成本”；准确表述是“无独立 API Key 与 API 账单，使用现有 Codex/ChatGPT 额度或活动 credits”。

## 7. 错误处理

- Notebook JSON 无效：停止并给出可操作的解析错误，不创建误导性报告。
- 依赖缺失：继续静态诊断，列出缺失包，不自动安装。
- 现有单元格报错：将错误作为诊断证据，不让单个错误中止整个流程。
- 执行超时：终止子进程，保留已完成步骤和超时位置。
- 风险扫描命中：阻止该次可选执行，仍允许静态诊断。
- 证据不足：显式标记，不推测未观察到的行为。
- 基线无法匹配：说明差异并请求确认，不自动比较两份不相关的 Notebook。
- 输出写入失败：先写临时文件再原子替换；不得覆盖已有 `baseline.json`。
- 多个基线匹配：列出 run ID 并要求明确选择，不自动采用最新运行。
- 挑战元数据缺失或 run ID 不匹配：源 Notebook 仍可复检，但挑战状态标记为“无法验证”，不猜测完成情况。
- 再次首次诊断同一 Notebook：创建新 `run_id` 目录，避免覆盖历史基线。

## 8. 测试与验收

### 8.1 测试分层

测试分为两层，避免为了测试而引入独立 API 调用：

- 本地自动化测试：解析、清洗、脱敏、风险扫描、run 选择、评分算术、临时执行、超时、哈希保护、产物结构和基线比较。使用固定 Notebook 与结构化诊断 fixture，不依赖模型或网络。
- GPT-5.6 会话评测：在 Codex 中使用安全样例，检查概念诊断、挑战质量和开放式答案复评。结果保存为预生成产物和人工验收记录，不需要 OpenAI API Key，也不作为每次本地测试的前提。

### 8.2 自动化验收场景

至少覆盖以下案例：

1. 正常 Notebook 生成全部必需产物，且结构校验通过；
2. 语法错误和已有运行时错误能定位到正确单元格；
3. 依赖缺失时静态诊断仍完成；
4. 无限循环在临时执行中按总时限终止；
5. 常见敏感值在模型上下文和产物中被脱敏；
6. shell、网络或危险文件系统操作阻止可选执行；
7. Base64 与巨大输出不会进入模型上下文；
8. 复检能识别已解决问题、回归和分数变化；
9. 工具运行前后源 Notebook 哈希保持不变；
10. 产物写入中断不会破坏已有基线；
11. 每项扣分都能追溯到单元格和规则；
12. 静态推断与实际执行结果在报告中有明确标签；
13. 多个匹配 run 时必须等待明确选择；
14. challenge 元数据能稳定关联两个任务与原 issue ID；
15. 首次诊断可选执行只运行源 Notebook 临时副本，并只更新目标隔离日志和 `report.md` revision；
16. 复检对 source 与 challenge 分别扫描、确认、执行和记录，且运行证据不会跨目标使用；
17. 任一可选执行都不修改基线、源 Notebook 或挑战 Notebook；
18. 同一次工具运行中检测到所选目标哈希变化时停止最终化。

### 8.3 安全演示样例

仓库提供 `samples/transformer_attention_buggy.ipynb`，只包含安全、确定且可快速演示的内容：

- 一个可定位的代码错误；
- 一个关于 attention 的概念误解；
- 一个可复现性问题。

样例应能让首次诊断、两个挑战和复检在五分钟内完成。仓库同时保留预生成的 `report.md`、`challenge.ipynb` 和 `verification.md`，使评委无需安装也能理解完整体验。

### 8.4 成功标准

- 新用户从全新 checkout 到生成第一份报告不超过五分钟；
- 默认路径不执行任何 Notebook 代码；
- 源 Notebook 在工具运行中从不被修改；
- 每个关键诊断都有单元格证据或“证据不足”标签；
- 两个挑战都能映射到已发现的问题；
- 复检始终关联用户明确选定或唯一匹配的 run；
- 复检展示可解释的前后变化，而非只给一个新分数；
- 核心演示可在三分钟视频内讲清楚。

## 11. 关键设计取舍

### 11.1 选择 Codex Skill，而非 Web 应用

Skill 最贴合活动要求，也能直接使用 GPT-5.6 与 Codex，无需另建 API 后端、认证、部署和计费链路。代价是体验依赖 Codex 客户端，无法提供独立网页界面；首版接受这一限制，以换取更低实现风险和更强的提交完整度。

### 11.2 静态分析优先，执行必须确认

完全自动执行更容易展示运行结果，但风险和环境不确定性过高。静态分析优先可保证默认安全和快速响应；受限执行只作为提高证据可信度的可选步骤。

### 11.3 分离报告与挑战 Notebook

把诊断写回源 Notebook 容易污染课程材料，也难以比较修改前后状态。独立报告便于评委阅读，独立挑战 Notebook 则保留可交互的学习体验，同时维持源文件不可变。

## 12. 完成定义

首版只有在以下事项全部满足时才算完成：

- Skill 可从全新 checkout 安装并调用；
- 静态解析、脱敏、风险扫描、产物生成和复检均有自动化测试；
- 可选执行具有确认、临时副本、超时和日志；
- 安全样例可完成完整演示闭环；
- README 准确覆盖安装、平台、成本、隐私和安全边界；
- 三分钟以内公开视频、公开仓库和 Devpost 必填字段准备完毕；
- 使用 GPT-5.6 的 Codex 会话及 `/feedback` Session ID 已记录；
- 提交前在受支持的全新环境完成一次端到端验证。
